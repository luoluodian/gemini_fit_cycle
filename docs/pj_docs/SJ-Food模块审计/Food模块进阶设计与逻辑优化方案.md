Food 模块进阶设计与逻辑优化方案 (Integrated Optimization Plan)

1. 数据架构与标准化 (Standardization)

1.1 字典化与维度锁定 (解决问题 1, 漏洞 1)

- 方案：将 category 和 unit 接入 data_dictionary。
- 关键改进：在 unit 字典中增加 dimension (维度) 字段。
  - 重量类 (weight): g, kg, oz
  - 体积类 (volume): ml, l, cup
  - 计数类 (count): piece, bar, scoop
- 逻辑约束：食材原型的 unit 决定其维度。饮食记录录入时，步进器的单位必须属于同一维度（例如：不能用“毫升”去量“鸡蛋”），从而避免 漏洞 1 的计算崩溃。

1.2 增强型标签系统 (解决问题 2, 3)

- 方案：支持 tags 多选，并引入“官方推荐标签”机制。
- 数据库变更：保留 JSON 存储，但在查询层支持 JSON_CONTAINS 过滤。
- 前端逻辑：分类 (category) 保持 1:1 用于视觉定义；标签 (tags) 1:N 用于精细化检索（如：高蛋白、低敏、素食）。

2. 公共化与社交治理 (Ownership & Social)

2.1 引用克隆机制 (解决问题 4, 5, 6, 漏洞 4)

- 方案：“逻辑下架”代替“物理删除” + “克隆引用”。
- 逻辑流程：
  1.  当用户 B 将用户 A 的公共食材加入计划时，系统记录引用关系。
  2.  若用户 A 删除该食材：
      - 若引用计数 > 0：食材进入“已下架”状态，对新用户不可见，但对已引用的用户 B 依然有效（解决漏洞 4 的僵尸数据问题）。
      - 若引用计数 = 0：执行软删除。
  3.  计划分享 (解决问题 6)：分享计划时，私有食材自动“提公”或生成副本，确保接收方数据闭环。

2.2 重复数据补偿 (解决问题 5)

- 方案：引入 “相似度算法 (Levenshtein Distance)”。
- 交互逻辑：当用户创建“鸡胸肉”时，系统检索库中已有的“系统级”或“高频公共级”类似食材，通过 Tip 提醒：“您是否在找：[官方]
  鸡胸肉？”，引导用户优先使用标准化数据。

3. 计算完整性与自动化 (Calculation & UX)

3.1 物理平衡校验 (解决漏洞 3)

- 后端强制约束 (Check Constraint)：
  - 公式：protein + fat + carbs <= baseCount。
  - 执行点：在 Create/UpdateFoodItemDto 提交时增加自定义校验器，若不符合直接返回 400 BadRequest。这从源头杜绝了“100g 食材含有 300g
    营养素”的逻辑悖论。

3.2 热量校验与快照同步 (解决漏洞 2)

- 方案：
  - 创建端：保留原有的 (P*4 + F*9 + C\*4) 热量偏差弹窗提醒。
  - 消费端 (解决漏洞 2)：当用户从“计划”转化为“打卡记录”时，若原食材数据已变动，系统提示：“食材营养数据已更新，是否同步至今日记录？”

3.3 自动化视觉辅助 (解决问题 7)

- 方案：关键词 Emoji 映射服务。
- 实现：维护一个 Map<string, string> 字典。
  - Map.get('肉') -> 🥩, Map.get('蛋') -> 🥚, Map.get('奶') -> 🥛
  - 若无匹配，则根据所属 category 自动分配。
