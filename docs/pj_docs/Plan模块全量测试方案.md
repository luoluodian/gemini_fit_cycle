## 2. 核心测试用例矩阵 (Core Test Matrix)

### 2.1 计划创建与碳循环算法 (Creation & Algorithm)
| 步骤 ID | 测试动作 | 预期结果 |
| :--- | :--- | :--- |
| **CA-1** | 选择“碳循环模式”，设置周期为 7 天 | UI 提示需分配 7 天的高/中/低碳日 |
| **CA-2** | 输入体重 70kg，系数 [2.0, 3.0, 0.8]，配置高碳日 | 验证算法生成的高碳日热量 = `70 * 2.0 * 系数` 等公式准确性 |
| **CA-3** | **配平校验**: 设置高+中+低碳日总和 = 6 天（少于周期 7 天） | “下一步”按钮禁用，并显示“还需配置 1 天”的黄色警告 |
| **CA-4** | 修改基础体重后进入下一步 | 观察所有已生成的日模板营养素目标是否同步按比例更新 |

### 2.2 状态排他性与生命周期 (Exclusivity & Lifecycle)

| **LC-3** | 手动点击“结束计划” | 计划状态即时更新，列表页将其移至“已完成”标签页 |

| 步骤 ID | 测试动作 | 预期结果 |

### 2.8 目标达成度与时间逻辑 (Goals & Timing)
| 步骤 ID | 测试动作 | 预期结果 |
| :--- | :--- | :--- |
| **GT-1** | 核对算法“建议目标值”与“实际添加食材总和” | UI 准确显示 Delta (差值)，如“还差 50kcal 达到目标”，且在差值超过 10% 时给出预警 |
| **GT-2** | **跨零点激活测试**: 在 23:59 分激活计划 | 验证次日凌晨记录模块是否能平滑拉取该计划，无 1 分钟内的生效延迟 |
| **GT-3** | **历史一致性约束**: 尝试修改已过去日期的模板数据 | 系统应限制或拦截修改，防止已发生的历史统计数据被由于模板变更导致紊乱 |

## 3. 极端值与鲁棒性测试 (Edge Cases)
- **极限周期**: 设置 1 天周期和 365 天周期（验证后端内存与循环性能）。
- **空餐次提交**: 创建一个没有任何食材的“空餐次”，验证 UI 渲染是否崩溃。
- **并发保护**: 在“确认计划”保存过程中模拟网络延迟，验证是否存在 Loading 遮罩拦截重复提交。

## 4. 故障判定标准
- **P0**: 数据库事务回滚失败导致计划天数与计划主体数据不一致。
- **P1**: 碳循环算法计算误差超过 5kcal。
- **P2**: 模板克隆功能失效或产生浅拷贝引用问题。
