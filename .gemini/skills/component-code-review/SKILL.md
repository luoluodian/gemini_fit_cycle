### 🚀 组件审计专家 Skill：完整定义 (V2.0)

你可以将此定义直接作为 Prompt 模版存入你的 AI 助手，或者作为开发文档规范。

#### 1. 核心审计矩阵 (The Master Matrix)

| 维度                      | 关键检查项 (Checkpoints)                            | 状态 | 审计细节与风险描述 |
| ------------------------- | --------------------------------------------------- | ---- | ------------------ |
| **1. 引用依赖 (Context)** | **依赖健康度**：是否引入了不必要的重型 npm 包？<br> |

<br>**影响范围**：修改此组件是否会导致其他页面崩溃？ | [ ] | |
| **2. 功能交互 (Logic)** | **业务逻辑**：核心交互（点击、跳转、表单）是否完备？<br>

<br>**状态覆盖**：Loading、Empty、Error 是否有兜底？ | [ ] | |
| **3. 数据契约 (Data)** | **鲁棒性**：对后端 API 的深层嵌套字段是否有可选链或默认值保护？<br>

<br>**Props 类型**：TS 接口是否严谨，是否有冗余 Props？ | [ ] | |
| **4. 性能健壮 (Stability)** | **小程序适配**：(Taro 特有) 是否存在频繁 `setData` 或过深组件嵌套？<br>

<br>**资源清理**：定时器、EventBus 监听是否在销毁阶段释放？ | [ ] | |
| **5. 可测试性 (Test)** | **测试友好度**：是否有关键的 `data-testid`？逻辑是否易于编写 Playwright/Jest 测试？ | [ ] | |
| **6. 规范实践 (Spec)** | **工程规范**：命名规范、Tailwind 兼容性、注释质量、圈复杂度。 | [ ] | |

---

#### 2. 代码坏味道与重构指令 (Actionable Output)

审计表格之后，必须强制输出以下三项，将审计转化为行动：

- **🚫 严重漏洞 (Critical Smells)**：
- _示例：组件内部直接操作了全局 window 对象，会导致 SSR 或小程序环境报错。_

- **🛠️ AI 重构方案 (Refactor Proposal)**：
- 提供一段优化后的代码片段（例如：将 300 行的巨石组件拆分为 `useComponentLogic` Hook 和 纯 UI 组件）。
